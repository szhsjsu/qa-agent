## 1. 系统负责人补充说明

### 1.1 上线后我会盯什么指标

不是 token 量或调用次数 —— 那是成本，不是质量。我会盯这五条：

| 指标 | 当前位置 | 异常处理 |
|---|---|---|
| answer_acc / cite_recall / refuse_acc **按 type 分桶** | 离线评测 10/10·10/10·4/4 | 总分会掩盖单一类型退步，必须分桶看 |
| **拒答率 vs 准确率的相对走势** | 当前 4/14 = 29% | 拒答率涨但准确率没涨 = 系统在"装聋"；要进生产再回滚 |
| **citation 后验校验失败率**（quote 在原文找不到的比例） | 未实现，下一步要做 | > 1% 即触发告警，模型在编引用 |
| **用户改写率**（同会话用户改 query 重问） | 未采集 | 改写率高 = 答得让用户不满意 |
| 延迟 p95 | 当前 5.3s | > 10s 必告警；自检是延迟主因，必要时改异步 |

每条都对应 `docs/design.md` 已经讨论过的设计，不是临时想的。

### 1.2 我会推进的下一步（按优先级）

| # | 工件 | 工作量 | 为什么先做这个 |
|---|---|---|---|
| 1 | manifest.json + 按 doc/version 增量索引 | 1 天 | 文档变化是上线第一周一定遇到的，现在是全量重建 |
| 2 | 结构化日志（structlog JSON + trace_id 串四步） | 半天 | 没日志没法迭代 bad case |
| 3 | citation 后验校验：LLM 写的 quote 必须在原文 substring | 半天 | 把"模型编引用"这个最容易的幻觉路径堵死 |
| 4 | 评测集按 type 分桶 + 与 history.csv 上一行 diff | 1 小时 | 防 prompt 回归 |
| 5 | bge-reranker 接进来跑评测对比 | 1 天 | top-1 命中率最便宜的天花板抬升 |

不优先做：Docker 化、reranker GPU 化、多文档大规模、前端 —— 这些都是有了真实流量后再说。

### 1.3 守住底线的硬规则（这几条不让步）

- **拒答优先于错答**。金融场景下"我找不到"比"瞎答一个数字"安全 100 倍。这也是我把 verifier 改宽时仍然保留三道闸（retrieval 阈值 / 生成自判 / 自检 high-risk）的原因。
- **prompt / 评测集 / 模型版本全部进 git**，不允许 hotpatch。`agent/prompts.py` 是代码不是配置。
- **改 prompt 必须跑评测集，退步直接 block**。`evals/history.csv` 的最新一行不允许低于上一行。
- **每个回答必须能 trace 到 page + chunk_id**。出错时 30 秒内能定位是哪个 chunk 引发的。
- **生产环境必须有人审通道**。`refused=true` 或 `confidence=low` 的回答，调用方必须有兜底逻辑或转人审。

### 1.4 跟上下游的接口约定

| 角色 | 约定 |
|---|---|
| 文档提供方（业务/客户） | PDF 必带元数据 `{doc_id, version, domain, valid_from}`，不接受散乱投递；过期文档我标 deprecated 而非物理删，保留 1 个版本回滚窗口 |
| 调用方（业务系统） | 接口契约 `{question, domain?, doc_id?}` → `{answer, citations, refused, confidence, trace_id}`；`refused=true` 时调用方必须有 fallback（默认文案 / 转人工） |
| 运维 | p95 > 10s / 拒答率突变 > 20% / 5xx > 1% / OCR 平均置信度跌破 0.85 → 触发 page |
| 我自己（每周） | 跑一遍评测、检查 history.csv 走势、把当周的 bad case 入候选评测集 |

## 2. 工作方式补充确认

### 2.1 与 AI 协作的边界（这道作业的真实做法）

**让 AI 做的**：
- 代码骨架（pyproject、CLI 命令脚手架、FastAPI、Pydantic schema）
- 样板实现（jieba+RRF 组合、JSON 解析与重试、CLI 表格渲染）
- 设计文档草稿（我再补真实数据反馈和取舍说明）
- 评测集第一版草稿（必须我对着 PDF 人核）

**不让 AI 做的**：
- 决定方向。"做不做 reranker"、"切块窗口多大"是我决定，让 AI 提选项不让 AI 拍板。
- 标评测集标准答案。**v1 评测集我让 AI 标 fact-1 的金标，结果错了**（把 2024-01-01 期初标成了 2024-12-31 期末），是我跑完评测发现 80% acc 时回去对 PDF 才发现的。这正好印证为什么不能让 LLM 自标自答。
- 接受未验证的 AI 建议。AI 说"这样写更好"必须用单测或评测验证，不验证不进 main。

**校验三件套**（写在 `docs/design.md §7`）：
- 重点模块（agent/pipeline.py 的判定逻辑、verifier 拒答阈值、chunker 切块规则）逐行看过；样板代码（CLI/FastAPI/dataclass）抽样看 —— 大约 6:4。
- 单测覆盖确定性逻辑（切块正则、表格 MD、query 改写）；评测覆盖端到端。
- 模型输出强制 JSON Schema，解析失败重试一次后降级拒答。

### 2.2 出 bug 时的定位流程（举本项目实例）

第一版评测 80%，fact-5 和 table-2 失败。我没有立刻改 prompt 或调阈值，走的是这套：

1. **不改代码，先复现**：`qa ask` 单独跑这两道题，确认稳定失败。
2. **沿管线分层定位**：
   - retrieval 层：top-3 包含正确 chunk → 检索没问题
   - 生成层：看模型 raw 输出，发现模型其实给出了正确答案
   - 自检层：verify 返回 `grounded=False` → **定位到 verifier 误杀**
3. **看 verifier 的实际输入**：发现两个问题
   - evidence 被截断到前 500 字，关键的"合计"行在 500 字之后
   - 拒答规则"grounded=False 就拒答"过严
4. **改最小一块**：只动 evidence 截断 + 拒答规则（`hallucination_risk=high` 才拒），不乱碰其他模块
5. **重跑全量评测**：80% → 100%，其他题没退步 → 进 git

这流程依赖三个习惯：(1) 分模块可观测；(2) 评测能定位到层；(3) 改动最小一块，单变量验证。我把这次复盘写进了 `docs/design.md §9`，不当遮掩。

### 2.3 评测集是和团队的共同语言

评测集不只是给自己用的回归工具，是跨角色对话的标准化容器：

- **产品提需求** → 翻译成 1-3 条评测题进 `qa.jsonl`，需求是否实现 = 这些题是否通过
- **客户反馈错答** → 该 case 当周入评测集，bug 修完后这条题永远必须过
- **模型 / prompt 升级** → 不能口头说"效果变好了"，要有 `history.csv` 的前后两行 + 按 type 分桶的 diff
- **跨部门同步** → 一张"按 type 分桶的准确率 + 拒答率"表，比讲架构图有用

这也是为什么我作业里 `evals/qa.jsonl` 标了 5 个 type（fact / table / no_answer / fuzzy / ocr_robust）—— 单一总分没法对话。

### 2.4 我对自己工作的评估方式

不是 "我提交了多少 PR / 写了多少行代码"，而是：

- 评测集覆盖的题型是不是真实业务的代表
- `history.csv` 这条线是不是在涨（即使涨得慢）
- 线上 bad case 进评测集的速度（理想：当周）
- prompt / 阈值改动是不是都有"为什么改"的 commit message
- 上线后出过错没 trace 不到的事故没有

如果一周下来：评测分数没动 / bad case 没沉淀 / commit 全是 "fix typo" / 出事故只能靠运气定位 —— 我自评不及格。
